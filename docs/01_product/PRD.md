# 1. บทสรุปผลิตภัณฑ์

Zayd คือผู้ช่วย AI สำหรับค้นคว้าและอธิบายความรู้อิสลามที่ออกแบบมาสำหรับผู้ใช้ในประเทศไทยโดยเฉพาะ ระบบจะตอบคำถามเป็นภาษาไทย พร้อมแสดงหลักฐาน แหล่งอ้างอิง ข้อความภาษาอาหรับ คำแปล และทัศนะทางฟิกฮ์ที่เกี่ยวข้องอย่างชัดเจน

Zayd ไม่ใช่ระบบออกฟัตวาอัตโนมัติ และไม่ควรถูกนำเสนอว่าเป็นมุฟตี AI แต่เป็นเครื่องมือช่วยค้นคว้า ทำความเข้าใจ และนำผู้ใช้ไปยังแหล่งข้อมูลที่ตรวจสอบได้

หัวใจของระบบประกอบด้วย:

1. ฐานข้อมูลความรู้อิสลามที่ตรวจสอบแล้ว
2. ระบบ Retrieval-Augmented Generation หรือ RAG
3. ระบบตรวจสอบแหล่งอ้างอิง
4. ระบบแยกทัศนะตามมัซฮับ
5. Reviewer Portal สำหรับผู้ตรวจทานและนักวิชาการศาสนา
6. ระบบจัดการลิขสิทธิ์และสิทธิ์การใช้ข้อมูล
7. ระบบส่งต่อคำถามเสี่ยงสูงให้ผู้รู้
8. ระบบประเมินคุณภาพคำตอบและป้องกัน hallucination
9. ระบบ Local-first RAG พร้อม External API fallback
10. ระบบติดตามย้อนหลังว่าคำตอบใช้เอกสาร โมเดล และ prompt เวอร์ชันใด

# 2. ปัญหาที่ต้องการแก้

## 2.1 ปัญหาของผู้ใช้ทั่วไป

ผู้ใช้มุสลิมในประเทศไทยพบปัญหาเมื่อค้นหาคำตอบเกี่ยวกับศาสนาอิสลาม เช่น:

- ข้อมูลภาษาไทยกระจัดกระจายอยู่หลายเว็บไซต์
- ไม่ทราบว่าแหล่งข้อมูลใดน่าเชื่อถือ
- คำตอบจาก AI ทั่วไปอาจสร้างหลักฐานหรือเลขหะดีษขึ้นเอง
- AI อาจผสมทัศนะจากหลายมัซฮับโดยไม่แจ้ง
- เนื้อหาส่วนใหญ่เป็นภาษาอาหรับหรืออังกฤษ
- คำตอบบางเรื่องไม่สอดคล้องกับบริบทของมุสลิมในประเทศไทย
- ผู้ใช้ไม่สามารถตรวจย้อนกลับว่าคำตอบมาจากหนังสือหรือแหล่งใด
- คำถามเกี่ยวกับการหย่า มรดก การเงิน และฮาลาลอาจมีความเสี่ยงสูงหากตอบผิด

## 2.2 ปัญหาของนักวิชาการและผู้ตรวจทาน

- ไม่มีระบบกลางสำหรับตรวจเอกสารก่อนนำเข้า AI
- ไม่มี workflow อนุมัติเนื้อหาศาสนาอย่างเป็นระบบ
- ไม่สามารถตรวจย้อนกลับว่า AI ใช้หลักฐานใด
- Feedback จากผู้ใช้ไม่ถูกจัดเป็นคิวสำหรับแก้ไข
- ขาดระบบ versioning และ audit log
- ไม่มี benchmark ภาษาไทยสำหรับวัดความสามารถของ Islamic AI

## 2.3 ปัญหาของผู้พัฒนาระบบ

- โครงการ Islamic AI ที่มีอยู่ส่วนใหญ่เน้นภาษาอาหรับหรืออังกฤษ
- โครงการ Open Source หลายโครงการไม่มี Reviewer Portal
- บางระบบพึ่งพา External API โดยไม่มีฐานข้อมูลในเครื่อง
- สิทธิ์ของ source code และสิทธิ์ของข้อมูลมักไม่ชัดเจน
- ระบบต้นแบบจำนวนมากไม่มีการวัด citation accuracy หรือ hallucination

# 3. วิสัยทัศน์ผลิตภัณฑ์

> ทำให้มุสลิมในประเทศไทยเข้าถึงความรู้อิสลามที่ตรวจสอบได้ เข้าใจง่าย และเคารพความแตกต่างทางวิชาการ โดยใช้ AI เป็นเครื่องมือช่วยค้นคว้า ไม่ใช่ผู้แทนของนักวิชาการศาสนา

# 4. Product Positioning

Zayd จะถูกวางตำแหน่งเป็น:

- ผู้ช่วยค้นคว้าความรู้อิสลามภาษาไทย
- เครื่องมือค้นหลักฐานจากอัลกุรอาน หะดีษ ตัฟซีร และฟิกฮ์
- ระบบอธิบายความเห็นของแต่ละมัซฮับ
- ระบบช่วยนักเรียนและครูสอนศาสนาค้นข้อมูล
- ระบบส่งต่อคำถามเฉพาะบุคคลให้ผู้รู้
- แพลตฟอร์มจัดการและตรวจสอบคลังความรู้อิสลามภาษาไทย

Zayd จะไม่ถูกวางตำแหน่งเป็น:

- มุฟตี AI
- ระบบออกฟัตวาโดยอัตโนมัติ
- ผู้แทนนักวิชาการศาสนา
- แหล่งตัดสินข้อพิพาททางศาสนา
- ระบบตัดสินว่าบุคคลใดเป็นผู้ศรัทธาหรือผู้ปฏิเสธศรัทธา
- ระบบให้คำวินิจฉัยทางกฎหมายหรือการแพทย์

# 5. เป้าหมายของผลิตภัณฑ์

## 5.1 เป้าหมายหลัก

1. ให้ผู้ใช้ถามคำถามเกี่ยวกับศาสนาอิสลามเป็นภาษาไทยได้
2. ให้คำตอบพร้อมหลักฐานที่ตรวจสอบย้อนกลับได้
3. ลดการสร้างข้อมูลและแหล่งอ้างอิงที่ไม่มีอยู่จริง
4. แยกทัศนะตามมัซฮับอย่างชัดเจน
5. ให้ผู้รู้ตรวจสอบเอกสารก่อนนำเข้าฐานความรู้
6. ให้ผู้ใช้รายงานคำตอบผิดหรือคลุมเครือได้
7. รองรับฐานข้อมูลแบบ Self-hosted และ External API fallback
8. สร้างคลังความรู้อิสลามภาษาไทยที่มี metadata และสิทธิ์การใช้งานชัดเจน
9. สร้าง benchmark ภาษาไทยสำหรับตรวจคุณภาพระบบ
10. รองรับการขยายไปยัง LINE, Mobile Application และ Public API

## 5.2 เป้าหมายที่ไม่ใช่เป้าหมายของ Zayd 1.0

- รองรับทุกมัซฮับอย่างครบถ้วน
- รองรับทุกหัวข้อศาสนา
- แทนที่นักวิชาการศาสนา
- ให้บริการฟัตวาเฉพาะบุคคลแบบอัตโนมัติ
- มีระบบมรดกและซะกาตเต็มรูปแบบ
- เปิด API สำหรับบุคคลภายนอกทันที
- รองรับภาษาไทย มลายู อาหรับ และอังกฤษอย่างสมบูรณ์พร้อมกัน
- รองรับผู้ใช้ระดับหลายล้านคนตั้งแต่วันแรก

# 6. กลุ่มผู้ใช้เป้าหมาย

## 6.1 มุสลิมทั่วไปในประเทศไทย

ความต้องการ:

- ถามคำถามด้วยภาษาไทยทั่วไป
- ได้คำตอบที่เข้าใจง่าย
- เห็นหลักฐานและคำแปล
- ทราบว่าคำตอบเป็นทัศนะของมัซฮับใด
- ตรวจสอบแหล่งที่มาได้

## 6.2 นักเรียนศาสนา

ความต้องการ:

- ค้นอายะฮ์ หะดีษ และข้อความจากหนังสือ
- เปรียบเทียบทัศนะ
- บันทึกคำตอบไว้ศึกษา
- เปิดต้นฉบับภาษาอาหรับ
- ดูข้อมูลผู้เขียน หนังสือ เล่ม และหน้า

## 6.3 ครูสอนศาสนาและอาจารย์

ความต้องการ:

- ใช้ระบบเป็นเครื่องมือค้นคว้า
- ตรวจแหล่งอ้างอิง
- ส่ง Feedback
- เสนอเอกสารเข้าสู่ระบบ
- ตรวจคำตอบที่ผู้ใช้รายงาน

## 6.4 ผู้เปลี่ยนมานับถือศาสนาอิสลาม

ความต้องการ:

- ภาษาไทยที่เข้าใจง่าย
- คำศัพท์ศาสนาพร้อมคำอธิบาย
- ขั้นตอนปฏิบัติพื้นฐาน
- หลีกเลี่ยงถ้อยคำที่ซับซ้อนเกินไป

## 6.5 ผู้ตรวจทานเนื้อหา

ความต้องการ:

- ตรวจเอกสารต้นฉบับและข้อความที่สกัด
- แก้ metadata
- ตรวจคำแปล
- ระบุมัซฮับ
- อนุมัติ ตีกลับ หรือปฏิเสธเอกสาร
- ตรวจประวัติการแก้ไข

## 6.6 นักวิชาการอาวุโส

ความต้องการ:

- ตรวจเรื่องที่มีความเห็นต่าง
- อนุมัติเนื้อหาฟิกฮ์
- ตรวจคำถามเสี่ยงสูง
- ระงับหรือแก้ไขคำตอบที่มีปัญหา
- กำหนดแนวทางการตอบของระบบ

## 6.7 ผู้ดูแลระบบ

ความต้องการ:

- จัดการผู้ใช้และบทบาท
- ตรวจสถานะระบบ
- จัดการแหล่งข้อมูล
- ดูต้นทุนและการใช้โมเดล
- ดูคิวงาน
- ตรวจ audit log
- ปิดการใช้งานแหล่งข้อมูลหรือโมเดลที่มีปัญหา

# 7. หลักการออกแบบผลิตภัณฑ์

## 7.1 Evidence First

ระบบต้องค้นหลักฐานก่อนสร้างคำตอบ หากหลักฐานไม่เพียงพอ ระบบต้องแจ้งอย่างตรงไปตรงมาและไม่เดา

## 7.2 Citation Required

ข้อกล่าวอ้างทางศาสนาที่สำคัญต้องมี citation จากเอกสารที่ระบบค้นพบจริง

## 7.3 Scholar in the Loop

เอกสารฟิกฮ์ คำแปล และเนื้อหาสำคัญต้องผ่าน workflow ตรวจสอบก่อนเผยแพร่

## 7.4 Madhhab Aware

ระบบต้องทราบว่าข้อมูลแต่ละชิ้นเป็นของมัซฮับใด และห้ามนำมาปะปนโดยไม่ระบุ

## 7.5 Thai Context

คำตอบต้องเหมาะกับผู้ใช้ในประเทศไทย รวมถึงบริบทภาษา กฎหมาย องค์กรอิสลาม และระบบฮาลาลไทย

## 7.6 Auditable

ทุกคำตอบต้องตรวจย้อนหลังได้ว่าใช้:

- โมเดลใด
- Prompt เวอร์ชันใด
- Retrieval เวอร์ชันใด
- เอกสารใด
- Citation ใด
- External API ใด
- กฎหรือ policy ใด

## 7.7 Privacy by Default

ข้อมูลผู้ใช้ต้องไม่ถูกนำไปฝึกโมเดลโดยอัตโนมัติ และผู้ใช้ต้องมีทางเลือกลบประวัติหรือใช้โหมดไม่บันทึกประวัติ

# 9. User Journey หลัก

## 9.1 การถามคำถามทั่วไป

```text
เปิด Zayd
→ พิมพ์คำถามภาษาไทย
→ ระบบวิเคราะห์ภาษา หัวข้อ ความเสี่ยง และมัซฮับ
→ ค้น Local RAG
→ เรียก External API หากจำเป็น
→ ตรวจความเพียงพอของหลักฐาน
→ สร้างคำตอบ
→ ตรวจ citation
→ แสดงคำตอบพร้อมแหล่งอ้างอิง
```

## 9.2 การเปิดดูหลักฐาน

```text
ผู้ใช้เปิดคำตอบ
→ กดแหล่งอ้างอิง
→ ดูข้อความต้นฉบับ
→ ดูภาษาอาหรับ
→ ดูคำแปลไทย
→ ดูชื่อหนังสือ เล่ม หน้า หรือเลขหะดีษ
→ ดูสถานะการตรวจสอบ
```

## 9.3 การรายงานคำตอบผิด

```text
ผู้ใช้กดรายงาน
→ เลือกประเภทปัญหา
→ เพิ่มคำอธิบาย
→ ระบบสร้าง Review Ticket
→ ผู้ตรวจได้รับคิว
→ ตรวจคำถาม คำตอบ และ Retrieval Trace
→ แก้ไขหรือปิด Ticket
```

→ สร้าง chunk และ embedding
→ Publish เข้า Production RAG

# 16. ระบบ Feedback และ Incident

## 16.1 ประเภท Feedback

- คำตอบผิด
- หลักฐานไม่ตรง
- เลขอายะฮ์ผิด
- เลขหะดีษผิด
- สถานะหะดีษผิด
- คำแปลผิด
- ตอบผิดมัซฮับ
- ผสมหลายทัศนะ
- ภาษาไม่เหมาะสม
- คำตอบอันตราย
- ควรส่งต่อผู้รู้
- อื่น ๆ

## 16.2 Incident Severity

### P0 — Critical

- คำตอบอาจก่อให้เกิดอันตรายร้ายแรง
- หลักฐานปลอมในหัวข้อสำคัญ
- ข้อมูลส่วนตัวรั่วไหล
- เอกสารต้องห้ามถูกเผยแพร่

### P1 — High

- คำตอบฟิกฮ์ผิดอย่างมีนัยสำคัญ
- ตอบผิดมัซฮับ
- Citation ไม่ตรง
- ระบบส่งคำตอบที่ควร abstain

### P2 — Medium

- คำแปลคลุมเครือ
- ภาษาไม่เหมาะสม
- Retrieval ไม่ครบ

### P3 — Low

- ปัญหา UI
- การสะกด
- Formatting

## 16.3 การแก้ไข

ทุก incident ที่ได้รับการยืนยันควรถูกเพิ่มเป็น regression test

# 18. Non-Functional Requirements

## 18.1 Performance

- หน้าเว็บหลักต้องโหลดได้ภายใน 3 วินาทีในเครือข่ายมือถือทั่วไป
- ระบบควรเริ่ม streaming คำตอบภายใน 5 วินาที
- คำตอบทั่วไปควรเสร็จภายใน 20 วินาที
- Retrieval ภายในควรเสร็จภายใน 2 วินาทีในภาวะปกติ

## 18.2 Availability

- เป้าหมาย MVP: 99.5%
- ระบบต้องแสดงข้อความที่เหมาะสมเมื่อ External API ใช้งานไม่ได้
- Local RAG ต้องยังทำงานได้เมื่อ provider ภายนอกล่ม

## 18.3 Scalability

ระบบต้องรองรับ:

- การเพิ่ม backend replica
- การเพิ่ม worker
- การแยก vector database
- การเพิ่ม provider
- การเพิ่มภาษา
- การเพิ่ม collection

## 18.4 Security

- TLS
- Encryption at rest
- RBAC
- MFA สำหรับ Admin และ Reviewer
- Rate limiting
- Audit log
- Secret management
- File malware scanning
- Dependency scanning
- Backup encryption
- Session timeout
- Protection against prompt injection

## 18.5 Privacy

- ไม่นำบทสนทนาไปฝึกโมเดลโดยค่าเริ่มต้น
- ผู้ใช้ลบประวัติได้
- มีโหมดไม่บันทึก
- ปกปิดข้อมูลส่วนบุคคลก่อนส่งไป provider เมื่อทำได้
- แยกข้อมูลผู้ใช้จาก corpus ความรู้
- กำหนด retention policy

## 18.6 Accessibility

- รองรับมือถือ
- รองรับตัวอักษรขนาดใหญ่
- Contrast เหมาะสม
- รองรับ screen reader ในส่วนหลัก
- ภาษาอาหรับต้องแสดง RTL ถูกต้อง
- ภาษาไทยต้องแสดง LTR ถูกต้อง

# 22. Success Metrics

1. ผู้ใช้ถามภาษาไทยและได้คำตอบพร้อมหลักฐาน
2. ผู้ใช้เปิดดูแหล่งอ้างอิงได้

5. Production RAG ใช้เฉพาะเอกสารที่ได้รับอนุมัติ

7. ผู้ใช้รายงานคำตอบผิดได้
8. ระบบ abstain เมื่อไม่มีหลักฐาน
9. ระบบส่งต่อคำถามเสี่ยงสูงอย่างเหมาะสม
10. ระบบทำงานได้เมื่อ External API บางรายล่ม
11. Backup และ Restore ผ่านการทดสอบ
12. ผ่าน benchmark และ Closed Pilot

# 23. ความเสี่ยง

## 23.1 ความเสี่ยงด้านข้อมูล

- การอนุมัติล่าช้า

- แบ่งระดับเนื้อหา

- ค่า LLM สูง
- External API มีค่าใช้จ่าย

# 25. Open Questions

หัวข้อต่อไปนี้ต้องตัดสินใจก่อนจัดทำ SRS ฉบับ Approved:

1. จะใช้มัซฮับชาฟิอีเป็นค่าเริ่มต้นแบบบังคับหรือให้เลือกตั้งแต่เริ่มใช้
2. นักวิชาการผู้อนุมัติจะมาจากองค์กรใด
3. จะรองรับฟัตวาจากหน่วยงานไทยใดบ้าง
4. จะเก็บประวัติผู้ใช้เป็นค่าเริ่มต้นหรือไม่
5. จะใช้ PostgreSQL + pgvector หรือ Qdrant ใน MVP
6. จะใช้โมเดลใดเป็นโมเดลหลักและโมเดลสำรอง
7. จะเปิด Guest mode โดยจำกัดจำนวนคำถามเท่าใด
8. จะมีระบบถามผู้รู้ใน MVP หรือหลัง MVP
9. จะรองรับเอกสารสแกนตั้งแต่ MVP หรือไม่
10. จะเปิด Source code ของ Zayd บางส่วนหรือทั้งหมดหรือไม่
11. จะให้บริการฟรีทั้งหมดหรือมีแพ็กเกจสนับสนุนค่าใช้จ่าย
12. จะเก็บ External API response ได้นานเท่าใดในแต่ละ provider

# 28. สรุปขอบเขต Zayd 1.0

Zayd 1.0 จะเป็น Mobile-first PWA สำหรับตอบและค้นคว้าความรู้อิสลามภาษาไทย โดยมีมัซฮับชาฟิอีเป็นค่าเริ่มต้น ใช้ Local-first RAG พร้อม External API fallback และมี Reviewer Portal สำหรับตรวจเอกสารก่อนเผยแพร่

ระบบจะเน้นความถูกต้อง การตรวจสอบย้อนกลับ และการแสดงข้อจำกัด มากกว่าการตอบทุกคำถาม

หลักสำคัญของ Zayd คือ:

> ไม่พบหลักฐานเพียงพอ ย่อมดีกว่าการสร้างคำตอบที่ดูน่าเชื่อถือแต่ตรวจสอบไม่ได้
